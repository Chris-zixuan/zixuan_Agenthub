# 简历优化器（Resume Optimizer）
## 核心工作流
本工具基于专业简历方法论（STAR法则、量化原则、ASK模型），提供完整的简历优化流程：
**执行流程**：输入模式识别 → 内容解析 → JD信息询问 → 信息补充交互 → 专业优化 → 逻辑质量检查 → 多格式输出 → 质量评估报告
**三种输入模式**：
- **模式A**：用户直接输入简历文本
- **模式B**：用户上传 Word 简历文档（.docx格式）
- **模式C**：用户上传 PDF 简历文档（文本型PDF，不支持扫描件）
**JD信息输入**：JD（职位描述）信息仅通过对话框输入，不支持文件上传

---
## 任务目标
1. 解析简历内容（文本/Word/PDF文档）并识别结构模块
2. 按专业规则优化各模块表述（基础信息、教育背景、工作经历、项目经历、技能证书）
3. 执行逻辑性、合理性、衔接性质量检查
4. 结合 JD 进行针对性匹配优化（如有）
5. 生成 Word 和 PDF 格式简历
6. 输出职场竞争力综合评分与质量检测报告

---

## 前置准备
### 目录结构
首次运行时自动创建：
```
<技能目录>/
├── input/     ← 用户上传的简历文件（支持 .docx/.pdf 格式）
└── output/    ← 优化后的简历和质量报告
    ├── <姓名>_简历.docx
    ├── <姓名>_简历.pdf
    └── <姓名>_简历质量检测报告.md
```
### 依赖说明
**依赖包**：
- `python-docx`：用于解析 Word 文档（.docx格式）
- `pdfplumber`：用于解析 PDF 文档（文本型PDF）
**限制说明**：
- 支持 .docx 和 .pdf 格式（不支持 .doc 格式）
- PDF仅支持文本型，不支持图片扫描件简历
- 针对中文简历优化
- 不执行联网搜索
**PDF生成依赖**：
- 优先使用 Microsoft Edge（Windows）
- 备选方案：Chrome/Chromium（跨平台）、wkhtmltopdf
- 降级方案：输出 HTML 文件，提示用户手动打印为 PDF

## 操作步骤
### 第一步：识别输入模式并确认 JD 需求
**输入模式**：
| 模式 | 触发条件 | 说明 |
|:---:|:---|:---|
| **A** | 用户直接输入简历文本 | 对文本语义分析，提取简历结构 |
| **B** | 用户上传 Word 简历（.docx）到 `./input/` 目录 | 使用 python-docx 读取并解析文档内容 |
| **C** | 用户上传 PDF 简历到 `./input/` 目录 | 使用 pdfplumber 读取并解析文档内容（仅支持文本型PDF） |

**判断规则**：
1. **文件查找位置**：优先在技能目录的 `./input/` 子目录查找，其次在当前运行目录的 `./input/` 子目录查找
2. 通过文件名区分：含"简历"/"resume"为简历文件
3. 通过文件扩展名自动识别格式：.docx → Word，.pdf → PDF
4. 无法区分时，使用 AskUserQuestion 询问用户
5. 找不到文件时，使用 AskUserQuestion 询问用户确认文件位置

**JD 信息补充询问**：
- 无论通过哪种模式获取简历后，**必须**使用 AskUserQuestion 询问用户：
  - 问题："是否需要补充岗位 JD（职位描述）信息？"
  - 选项：["是", "否"]
- 如果用户选择"是"，引导用户在对话框输入 JD 文本内容（JD信息仅通过对话框输入，不支持文件上传）
- 如果用户选择"否"，则按常规规则进行优化

### 第二步：读取与解析简历内容
**模式A（文本输入）**：
- 直接对用户输入进行语义分析
- 识别并提取简历各模块
- 信息不足时，使用 AskUserQuestion 引导补充

**模式B（Word文档输入）**：
- 运行 `scripts/read_resume.py` 读取 Word 文档
- 脚本使用 python-docx 库解析 .docx 文件

**模式C（PDF文档输入）**：
- 运行 `scripts/read_resume.py` 读取 PDF 文档
- 脚本使用 pdfplumber 库解析文本型PDF文件
- 对提取的文本进行语义分析，识别简历各模块

**JD 信息处理（如有）**：
- 如果用户通过对话框提供了 JD 信息
- 使用 ASK 模型分析 JD 需求：
  - **A**ttitude：岗位要求的态度/软素质
  - **S**kill：技能/技术栈
  - **K**nowledge：知识/行业经验
- 提取 JD 关键词用于匹配

### 第三步：逐模块分析与信息补充
对简历各模块进行分析，判断是否需要补充信息。
**分析重点**：
| 模块 | 检查要点 |
|:---:|:---|
| 基础信息 | 是否有个人概况（50-80字）？联系方式是否完整？ |
| 教育背景 | 是否倒序？是否有亮点（GPA/排名/奖项）？ |
| 工作经历 | 是否使用 STAR 法则？是否有量化数据？动词是否专业？ |
| 项目经历 | 是否有典型项目？结构是否完整？成果是否量化？ |
| 技能证书 | 是否与岗位相关？是否有佐证？ |
**信息补充规则**：
| 优先级 | 情况 | 处理方式 |
|:---:|:---|:---|
| 高 | 工作经历完全无量化数据 | 必须提示补充 |
| 高 | 项目经历无成果描述 | 必须提示补充 |
| 中 | 工作经历描述模糊 | 建议提示补充 |
| 中 | 缺少个人概况 | 自动生成，不提示 |
| 低 | 教育背景缺少 GPA/排名 | 可选提示 |
| 低 | 其他信息过于简略 | 可选提示 |

> ⚠️ **重要**：使用 AskUserQuestion **一次性**列出所有需要补充的信息，避免反复询问。

### 第四步：执行简历优化
按照 `references/optimization-rules.md` 中的规则进行优化。
#### 核心原则
1. **绝对保留所有细节**：工作经历、项目经历的所有细节性描述必须完整保留，不得删除或忽略任何具体信息，包括但不限于：技术细节、业务场景、解决过程、项目实施细节、产品设计细节、团队协作过程、沟通协调细节、决策过程等，这是简历真实性的关键
2. **优化限于表达**：优化仅限于文字组织和表达层面，使语句更顺畅、专业，提升可读性
3. **细节补充完善**：如果用户提供的工作经历、项目经历缺乏细节性描述（包括技术和非技术性细节），必须使用大模型能力进行合理补充，使简历内容更真实、更具体
4. **基于事实发挥**：所有扩写和补充必须基于用户输入的信息，保持真实性和可信度，禁止编造未提及的技能或经历

#### 优化策略
| 情况 | 策略 |
|:---|:---|
| 有详细技术细节 | 完整保留，优化语句流畅度 |
| 有业务场景描述 | 完整保留，可适当丰富背景信息 |
| 有问题解决过程 | 完整保留，可扩展解决方案细节 |
| 有项目实施细节 | 完整保留，可丰富实施过程描述 |
| 有产品设计细节 | 完整保留，可扩展设计思路和决策过程 |
| 有团队协作描述 | 完整保留，可补充协作方式和沟通细节 |
| 描述过于简略 | 使用大模型能力合理补充细节（技术栈、团队规模、时间跨度、具体操作、实施过程、设计细节等） |
| 缺乏场景描述 | 补充合理的工作场景和背景信息 |
| 缺乏技术细节 | 基于已有信息推断并补充合理技术细节 |
| 缺乏非技术细节 | 基于已有信息补充合理的实施过程、设计细节、协作过程等 |
| 有具体数据 | 保留原数据，优化表述方式，丰富细节描述 |
| 用户补充了数据 | 使用补充数据进行 STAR 改写，扩展完整情境描述 |
| 用户无法补充 | 使用定性描述替代，突出行动复杂度，合理扩写背景信息 |
| 描述模糊 | 在原意范围内专业化改写，补充合理细节使内容更丰富 |
| 内容简略 | 基于输入内容进行合理扩写，丰富工作场景、行动细节和成果描述 |
| 缺少个人概况 | 根据经历自动生成 50-80 字 |
| 有 JD 输入 | 优先突出与 JD 匹配的内容，针对性扩展相关经历 |

#### 行业差异化优化
根据目标岗位类型采用不同优化策略：
| 岗位类型 | 优化重点 |
|:---|:---|
| 技术类 | 突出技术栈深度、性能优化成果、架构设计能力 |
| 产品类 | 强调用户思维、数据分析、产品从0到1经验 |
| 运营类 | 突出活动策划、用户增长、ROI等量化指标 |
| 销售类 | 强调业绩成果、客户资源、回款率 |
| 职能类 | 突出流程优化、成本控制、合规管理 |
| 设计类 | 展示作品集、用户体验优化成果 |
| 市场类 | 强调品牌传播、媒体关系、活动效果 |

#### JD 针对性优化（如有 JD 信息）
当用户通过对话框提供了 JD 信息时，执行以下优化：
1. **关键词匹配**：将 JD 关键词自然融入简历内容
2. **经历排序**：将与 JD 最相关的经历前置展示
3. **技能突出**：优先展示 JD 要求的技能和技术栈
4. **概况定制**：根据 JD 重新生成个人概况，突出匹配点
5. **内容强化**：针对 JD 要求的能力，强化相关经历描述

### 第五步：简历逻辑质量检查
对简历进行全面的逻辑性、合理性、衔接性检查。

**检查维度**：
| 检查项 | 检查内容 | 问题等级 |
|:---:|:---|:---:|
| **时间衔接性** | 教育/工作经历时间是否衔接，有无空缺期 | 高 |
| **职业连贯性** | 工作经历有无领域重大变动 | 中 |
| **项目一致性** | 不同项目是否存在重大异质性 | 中 |
| **角色稳定性** | 项目中承担工作是否重大变动 | 中 |

**检查流程**：
1. **时间衔接性**：提取时间线，检查空缺期。超过 3 个月需询问原因
2. **职业连贯性**：分析领域和内容，跨领域变动需询问原因
3. **项目一致性**：检查技术栈/业务差异，差异大需询问
4. **角色稳定性**：检查角色变化，跳跃/倒退需询问原因

**问题处理**：记录问题 → 询问用户（AskUserQuestion 一次性列出）→ 等待回复 → 润色补充

> ⚠️ **重要**：润色必须基于用户真实回复，不得编造。

### 第六步：生成输出文件

#### 6.1 生成 Word 简历
- 运行 `scripts/generate_docx.py` 脚本
- 脚本使用标准库直接生成 .docx 文件，无需安装 python-docx
- **零依赖**：仅使用 Python 标准库
- **A4 打印规范**：设置标准 A4 纸张大小（210mm × 297mm）
- **标准页边距**：上下 2.54cm，左右 3.17cm
- **专业样式**：包含页眉（简历）、页脚（页码）、多级标题样式
- 保存至 `./output/<姓名>_简历.docx`（如有JD信息：`./output/<姓名>_简历_<目标岗位>.docx`）

#### 6.2 生成 PDF 简历
- 运行 `scripts/generate_pdf.py` 脚本
- 脚本使用标准库将内容转换为 HTML，再调用浏览器打印功能生成 PDF
- **零依赖**：无需安装 weasyprint 或 pdfkit
- 保存至 `./output/<姓名>_简历.pdf`（如有JD信息：`./output/<姓名>_简历_<目标岗位>.pdf`）
**PDF 生成失败处理**：
- 如果系统不支持直接生成 PDF，输出 HTML 文件并提示用户手动打印为 PDF

### 第七步：生成简历质量检测报告

采用「职场竞争力评估体系」，从 7 个维度加权评分：
| 评估维度 | 权重 | 评估重点 |
|:---:|:---:|:---|
| 项目业绩 | 25% | 项目数量、规模、角色、成果量化、STAR完整性 |
| 工作成果 | 25% | 成果量化、显著性、行动描述、专业表达 |
| 个人技能 | 15% | 技能相关度、深度、广度、佐证 |
| 资质证书 | 10% | 证书含金量、相关性、语言能力 |
| 简历逻辑性 | 10% | 职业发展逻辑、能力递进性、内容一致性 |
| 简历合理性 | 10% | 经验年限匹配、岗位职责匹配、成果匹配 |
| 简历衔接性 | 5% | 时间衔接、领域衔接、技能衔接、角色衔接 |

**评级标准**：
| 得分 | 评级 | 含义 |
|:---:|:---:|:---|
| 90-100 | S | 极具竞争力 |
| 80-89 | A | 竞争力强 |
| 70-79 | B | 具备竞争力 |
| 60-69 | C | 竞争力一般 |
| 50-59 | D | 竞争力较弱 |
| 0-49 | E | 竞争力不足 |

**评分原则**：
1. **客观性**：基于简历明确呈现的信息
2. **保守性**：信息不足时给予中等偏下评分（50-65 分）
3. **建设性**：每个维度给出优点、不足、优化建议
4. **区分度**：体现不同简历的质量差异

**生成报告文件**：
- 使用 `assets/quality-report-template.md` 模板
- 保存至 `./output/<姓名>_简历质量检测报告.md`

### 第八步：输出优化说明

向用户展示：
1. **优化摘要**：整体优化了哪些方面
2. **各模块优化说明**：每个模块具体改动
3. **逻辑质量检查结果**：时间衔接性、职业连贯性等结论
4. **质量检测报告摘要**：综合得分、评级、核心优势、主要短板
5. **信息补充记录**：提示用户补充的内容及回复
6. **润色补充记录**：根据用户回复进行的润色
7. **未修改说明**：信息不足仅做语词修饰的部分
8. **文件链接**：提供简历 Word、PDF、质量检测报告的下载链接

## 注意事项
1. **真实性第一**：所有优化必须在用户真实信息基础上进行
2. **不执行联网搜索**：所有规则来自本地参考文档
3. **信息不足时**：优先提示用户补充，无法补充时才进行语词修饰
4. **逻辑检查必做**：第五步不可跳过
5. **质量报告必做**：第七步不可跳过
6. **发现问题主动询问**：不能自行编造解释
7. **评分客观保守**：信息不足时给予中等偏下评分
8. **PDF 降级方案**：系统不支持时输出 HTML 提示用户手动转换
9. **中文支持**：确保中文字体正确渲染
10. **文件覆盖保护**：输出目录存在同名文件时询问用户

## 使用示例

### 示例三：结合 JD 优化

2. 读取并解析简历内容
3. 使用 AskUserQuestion 询问是否需要补充 JD 信息
4. 用户选择"是"，在对话框输入 JD 文本内容
5. 使用 ASK 模型分析 JD 需求
6. 逐模块分析，突出与 JD 匹配内容
7. 执行逻辑质量检查
8. 生成针对性优化的 Word 和 PDF
9. 生成质量检测报告（结合 JD 评估匹配度）
10. 输出优化说明和文件链接

## 错误处理

| 错误场景 | 处理方式 |
|:---|:---|
| Word 文档读取失败 | 提示用户检查文件格式是否为 .docx，或建议直接粘贴文本 |
| PDF 生成失败 | 输出 HTML 文件，提示用户用浏览器打开并打印为 PDF |
| 用户输入信息极度不足 | 提示用户提供最小必要信息（姓名、联系方式、至少一段经历） |
| 发现严重逻辑矛盾 | 使用 AskUserQuestion 询问用户核实 |
| 输出文件已存在 | 询问用户是否覆盖，或建议修改文件名 |
| 无可用浏览器生成 PDF | 输出 HTML 文件，提示用户手动转换 |
| 用户拒绝补充信息 | 基于现有信息进行优化，在质量报告中注明信息不足 |
| JD 信息无法匹配 | 按常规规则优化，不进行针对性匹配 |
