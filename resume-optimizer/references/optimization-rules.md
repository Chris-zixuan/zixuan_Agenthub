# 简历优化规则

## 核心原则

**核心原则必须严格遵守**：

1. **绝对保留所有细节**：工作经历、项目经历的所有细节性描述必须完整保留，不得删除或忽略任何具体信息，包括但不限于：技术细节、业务场景、解决过程、项目实施细节、产品设计细节、团队协作过程、沟通协调细节等，这是简历真实性的关键
2. **优化限于表达**：优化仅限于文字组织和表达层面，使语句更顺畅、专业，提升可读性
3. **细节补充完善**：如果用户提供的工作经历、项目经历缺乏细节性描述（包括技术和非技术性细节），必须使用大模型能力进行合理补充，使简历内容更真实、更具体
4. **基于事实扩写**：所有扩写和补充必须基于用户输入的信息，保持真实性和可信度，禁止编造未提及的技能或经历

## 一、简历核心结构

| 模块 | 重要性 | 内容要求 |
|:---|:---:|:---|
| **基础信息** | 高 | 姓名、联系方式、个人概况（50-80字）、求职意向 |
| **教育背景** | 中 | 学历、专业、GPA（优秀者）、相关课程/项目 |
| **工作/实习经历** | 极高 | 核心模块，详细描述职责与成果 |
| **项目经历** | 高 | 突出项目经验、技术能力和个人贡献 |
| **其他信息** | 中 | 技能、证书、语言能力、荣誉奖项 |

**结构原则**：
- 信息层级清晰，重点突出
- 时间倒序排列（最新经历在前）
- 重点内容前置

## 二、各模块优化规则

### 2.1 基础信息优化

**优化目标**：简洁、专业、有吸引力
**必保留项**：姓名、联系电话、电子邮箱  
**可选保留项**：求职意向、现居城市、个人作品集链接、LinkedIn、GitHub  
**应删除项**：身高体重、婚姻状况、身份证号、政治面貌、照片（除非岗位明确要求）

**个人概况生成规则**（如用户未提供）：
- 第一句话点明身份：X年 + 领域/岗位 + 经验
- 提炼2-3个最突出的能力或量化成果
- 严格控制在50-80字

**示例**：
```
5年互联网产品经理经验，专注用户增长与商业化。主导过DAU从0到100万的产品，擅长数据驱动决策。
```

### 2.2 教育背景优化
**优化规则**：
1. 按时间倒序排列（最高学历在前）
2. 统一格式：`起止时间    学校名称    专业    学历`
3. 有工作经验者保持简洁（1-2行）
4. 应届生可适当展开相关课程、学术项目

**GPA处理规则**：
- GPA ≥ 3.5/4.0：保留并突出
- 3.0 ≤ GPA < 3.5：可选保留
- GPA < 3.0：建议删除

### 2.3 工作/实习经历优化（核心模块）
**优化目标**：**完整保留所有细节** + STAR法则 + 量化成果 + 专业表达 + 细节补充

#### 核心原则
**细节至上**：用户提供的所有细节描述必须100%保留，包括但不限于：技术细节、业务场景、解决过程、具体操作、项目实施细节、产品设计细节、团队协作过程、沟通协调细节、决策过程等，这是面试官判断简历真实性的首要因素

#### STAR法则深化应用

| 步骤 | 说明 |
|:---:|:---|
| **S-情境** | 保留并丰富工作背景和挑战描述 |
| **T-任务** | 精确理解用户提到的目标或职责 |
| **A-行动** | **完整保留所有细节**，使用专业动词优化表达，缺乏细节时主动补充（技术和非技术细节） |
| **R-成果** | 量化结果并说明业务价值，无数据时用定性描述 |

#### 细节保留与补充规则
| 情况 | 处理方式 |
|:---|:---|
| 有详细技术细节 | 完整保留，优化语句流畅度 |
| 有业务场景描述 | 完整保留，可适当丰富背景信息 |
| 有问题解决过程 | 完整保留，可扩展解决方案细节 |
| 有项目实施细节 | 完整保留，可丰富实施过程描述 |
| 有产品设计细节 | 完整保留，可扩展设计思路和决策过程 |
| 有团队协作描述 | 完整保留，可补充协作方式和沟通细节 |
| 描述过于简略 | 使用大模型能力合理补充细节（技术栈、团队规模、时间跨度、具体操作、实施过程、设计细节等） |
| 缺乏场景描述 | 补充合理的工作场景和背景信息 |
| 缺乏技术细节 | 基于已有信息推断并补充合理技术细节 |
| 缺乏非技术细节 | 基于已有信息补充合理的实施过程、设计细节、协作过程等 |

#### 量化原则
- **有数据时**：直接使用用户提供的数字，优化表述方式
- **无数据时**：使用定性描述替代，禁止编造具体数字

**量化维度**：数量（用户数、订单量）、效率（时间缩短、成本降低）、质量（转化率、留存率）、影响力（收入、增长）

#### 表达方式多样化规则

**核心原则**：动词应自然融入工作经历描述，避免机械地将动词放在句首。

**多样化表达结构**：

| 结构类型 | 示例 |
|:---|:---|
| 动词开头 | 主导产品从0到1的全流程设计与落地 |
| 成果/数据开头 | 实现用户转化率提升30%，通过优化推荐算法 |
| 场景/背景开头 | 在高并发场景下，设计分布式缓存方案 |
| 职责/角色开头 | 作为技术负责人，统筹团队完成核心系统重构 |
| 方法/策略开头 | 通过数据驱动的运营策略，推动用户增长 |
| 问题/挑战开头 | 面对用户留存下滑问题，设计分层运营方案 |

**注意事项**：
1. 同一段经历中，同一个动词不要出现2次以上
2. 优先使用实动词，避免虚词过度使用
3. 轮换使用不同的表达结构，避免单调重复
4. **核心原则**：表达流畅自然 > 形式统一

#### 分点陈述规则
- 每段工作经历列出3-5个要点
- 每个要点可根据内容丰富程度适当扩展
- 每个要点展示一项核心能力及详细背景
- 按重要程度排序（最重要的放第一个）
- 遵循"行动 + 细节 + 成果"的逻辑结构

**优化前后对比**：

| 优化前 | 优化后 |
|:---|:---|
| 负责产品运营工作 | 统筹产品全生命周期运营，从用户获取到留存变现全链路管理，推动日活用户增长120% |
| 参与用户增长项目 | 主导用户增长项目，设计并实施A/B测试方案，通过精准推送策略提升转化率28% |

### 2.4 项目经历优化
**优化规则**：
1. **绝对保留所有细节**：保留所有用户提及的项目和细节描述，不删减任何技术细节、业务场景、解决过程、项目实施细节、产品设计细节、团队协作过程等
2. 每个项目按STAR结构组织，**保留原有所有细节**并丰富背景描述
3. 项目信息包含：项目名称、时间、角色、背景、目标、行动、成果、技术细节、实施细节、设计细节
4. **完整保留技术难点和解决方案**，详细描述解决过程，缺乏细节时主动补充（技术和非技术细节）
5. 量化项目成果和影响，如无数据可合理扩写定性描述

**核心原则**：用户提供的技术栈、代码片段、架构设计、问题解决过程、项目实施细节、产品设计细节、团队协作过程等所有细节必须完整保留，这是展示技术能力、项目真实性和个人贡献的关键

**项目信息扩展要点**：
- **项目背景**：为什么做这个项目？遇到什么挑战？（保留并丰富）
- **技术栈**：使用了哪些技术？为什么选择这些技术？（保留并可补充技术选型理由）
- **团队协作**：团队规模、角色分工、协作方式、沟通协调过程（保留并可合理补充）
- **难点解决**：遇到什么技术或业务难点？如何解决？（完整保留，缺乏细节时补充）
- **成果影响**：项目带来了什么价值？对业务有什么影响？（保留并可扩展）
- **技术细节**：核心实现、关键代码、架构设计、优化过程（完整保留）
- **实施细节**：项目实施步骤、时间计划、资源调配、风险管理（完整保留，缺乏时补充）
- **设计细节**：产品设计思路、用户体验考量、交互设计细节（完整保留，缺乏时补充）

**细节补充规则**：
| 情况 | 处理方式 |
|:---|:---|
| 有详细技术实现 | 完整保留，优化表述流畅度 |
| 有架构设计描述 | 完整保留，可补充设计思路 |
| 有问题解决过程 | 完整保留，可扩展技术细节 |
| 有项目实施细节 | 完整保留，可丰富实施过程描述 |
| 有产品设计细节 | 完整保留，可扩展设计思路和决策过程 |
| 有团队协作描述 | 完整保留，可补充协作方式和沟通细节 |
| 描述过于简略 | 使用大模型能力合理补充技术细节、业务场景、实施过程、设计细节等 |
| 缺乏技术细节 | 基于已有信息推断并补充合理技术实现细节 |
| 缺乏非技术细节 | 基于已有信息补充合理的实施过程、设计细节、协作过程等 |

**示例**：
```
【电商平台核心系统开发】| 2020.03 - 2021.06 | 后端技术负责人
• 项目背景：公司原有系统架构无法支撑百万级并发
• 技术选型：采用微服务架构，使用Spring Cloud + Docker + Kubernetes
• 核心职责：带领8人后端团队，负责订单、支付、库存三大核心模块
• 技术难点：解决分布式事务一致性问题，通过Seata实现最终一致性方案
• 项目成果：系统吞吐量提升300%，订单成功率从95%提升至99.9%
```

### 2.5 其他信息优化

**优化规则**：
1. 只保留与岗位相关的技能和证书
2. 删除过于基础的技能（如Word、Excel等通用技能）
3. 技能按熟练度或与岗位相关度排序
4. 语言能力注明等级（如CET-6、雅思6.5）
5. 证书注明颁发机构和时间（如PMP、CFA）

**技能表述优化**：
- 普通：熟练使用Java → 优秀：精通Java/JVM体系，深入理解JVM调优和性能优化

## 三、信息不足处理规则

### 3.1 信息补充优先级
| 优先级 | 情况 | 处理方式 |
|:---:|:---|:---|
| 高 | 工作经历完全没有量化数据 | 必须提示补充 |
| 高 | 项目经历没有成果描述 | 必须提示补充 |
| 中 | 工作经历描述模糊 | 建议提示补充 |
| 中 | 缺少个人概况 | 自动生成，不提示 |
| 低 | 教育背景缺少GPA/排名 | 可选提示 |
| 低 | 其他信息过于简略 | 可选提示 |

### 3.2 用户无法补充时的处理

1. **保留原始描述的核心信息**，不删减用户已有的内容
2. **优化表述方式**，使用更专业的语言
3. **使用定性描述**替代量化
4. **突出行动本身的复杂度和规模**
5. **合理扩写**：基于用户输入进行合理扩写，丰富工作场景、技术细节和成果描述

### 3.3 内容扩写规则

**扩写原则**：
1. **基于事实**：所有扩写内容必须基于用户输入的信息
2. **合理推断**：根据行业常识和逻辑进行合理推断和补充
3. **丰富细节**：补充工作场景、技术栈、团队规模、时间跨度等
4. **深化描述**：扩展问题解决过程、技术难点、解决方案细节
5. **保持专业**：使用专业术语和表达方式提升简历质量

**扩写技巧**：
- **场景化扩展**：描述工作环境、团队规模、项目背景
- **过程化扩展**：描述任务执行的具体步骤和方法
- **成果化扩展**：描述工作带来的影响和价值
- **专业化扩展**：使用专业术语和行业黑话

**扩写示例**：
- 原始：负责项目开发
- 扩写：主导XX项目的全周期开发工作，从需求分析、技术选型到架构设计全程参与，带领5人团队完成核心模块开发，确保项目按时交付

**注意事项**：
- 禁止添加用户未提及的技能或经历
- 禁止编造具体数据（如具体数字、百分比）
- 扩写应服务于提升简历专业性和丰富性

## 四、结合JD优化规则

### 4.1 JD深度分析（ASK模型）

- **A**ttitude：岗位要求的态度/软素质（团队协作、沟通能力、抗压能力）
- **S**kill：岗位要求的技能/技术栈（Python、Java、项目管理）
- **K**nowledge：岗位要求的知识/行业经验（电商、金融、医疗）

### 4.2 简历匹配优化策略

| 策略 | 操作 |
|:---|:---|
| **关键词融入** | 将JD关键词自然融入简历各模块 |
| **经历排序** | 将与JD最相关的经历前置 |
| **技能优先级** | 优先展示JD要求的技能 |
| **个人概况重写** | 根据JD定制个人概况 |
| **成果对齐** | 强调与JD要求相关的成果 |

### 4.3 匹配度评估

| 匹配度 | 标准 |
|:---|:---|
| 高度匹配 | 技能匹配度≥80%，相关经验≥3年 |
| 中度匹配 | 技能匹配度50%-80%，相关经验1-3年 |
| 低匹配 | 技能匹配度<50%，相关经验<1年 |

## 五、语词修饰规则

### 5.1 语言风格指南

**高端简历语言特征**：
- 使用书面化、专业化的表达
- 避免口语化表达和主观评价
- 使用简洁、有力的短句
- 保持客观中立的语气

**专业表达替换**：
- 口语：我觉得 → 专业：经分析认为
- 口语：弄好了 → 专业：完成交付
- 口语：挺好的 → 专业：成效显著

### 5.2 修饰程度界定

| 修饰级别 | 定义 |
|:---:|:---|
| 轻度修饰 | 优化句子结构，替换非专业用词 |
| 中度修饰 | 重新组织逻辑，使用STAR结构 |
| 深度修饰 | 合理扩写，丰富细节描述 |
| 重度修饰 | 禁止！不得改变原始信息的含义 |

### 5.3 禁止事项清单
- 禁止添加用户未提及的工作内容
- 禁止编造量化数据
- 禁止夸大用户的工作成果
- 禁止添加用户未掌握的技能
- 禁止改变用户工作经历的时间线
- 禁止修改用户的核心信息

## 六、简历逻辑质量检查规则

### 6.1 时间衔接性检查

| 空缺位置 | 允许时长 | 超时处理 |
|:---|:---:|:---|
| 教育与工作之间（应届生） | ≤3个月 | 正常求职期 |
| 工作与工作之间 | ≤3个月 | 正常求职/休息 |
| 工作与工作之间 | >3个月 | **必须询问用户原因** |

### 6.2 职业连贯性检查

| 变动类型 | 处理方式 |
|:---|:---|
| 同领域晋升 | 正常 |
| 同领域平调 | 正常 |
| 跨领域变动 | **需询问变动原因** |
| 行业跨度大 | **需询问变动原因** |

### 6.3 项目一致性检查

| 异质性表现 | 处理方式 |
|:---|:---|
| 技术栈差异大 | **需询问是否真实** |
| 业务领域差异 | **需询问项目背景** |
| 角色差异大 | **需确认角色变化原因** |

### 6.4 角色稳定性检查

| 变化类型 | 判断 |
|:---|:---|
| 能力递进 | 正常成长轨迹 |
| 角色跳跃 | **需确认真实性** |
| 能力倒退 | **需询问原因** |

## 八、高级优化技巧

### 8.1 关键词优化
- **JD关键词匹配**：提取JD中的核心关键词，自然融入简历
- **行业术语**：使用目标行业的专业术语和黑话
- **动词多样性**：避免重复使用相同的动词
- **数字优先**：量化成果优先展示

### 8.2 排版优化
- **字体选择**：使用专业字体（微软雅黑、PingFang）
- **字号层级**：标题>正文>辅助信息
- **留白适度**：段落间距合理，避免过于拥挤
- **重点突出**：使用加粗突出关键数据和成果

### 8.3 ATS优化
- **关键词密度**：合理分布关键词，避免堆砌
- **标准格式**：使用标准标题和列表结构
- **文件格式**：提交PDF格式
- **文件名规范**：姓名_岗位_城市.pdf

### 8.4 简历长度控制

| 经验年限 | 建议页数 |
|:---|:---:|
| 0-2年 | 1页 |
| 3-5年 | 1-2页 |
| 5年以上 | 2页 |
